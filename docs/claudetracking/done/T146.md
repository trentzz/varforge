# T146: Implement inline UMI FASTQ output with optional spacer

**Epic**: EPIC-KAM-COMPAT
**Priority**: P1
**Depends on**: none
**Status**: done

## Goal

When `umi.inline: true`, prepend the UMI (and optional spacer) to the start
of both R1 and R2 read sequences in FASTQ and BAM output. This matches the
Twist FASTQ layout (`5M2S+T`) that kam expects.

## Success Criteria

- [ ] `UmiConfig` has new field `spacer: Option<String>`, deserialisable from YAML.
- [ ] Validation no longer rejects `umi.inline: true` (remove the T134 guard).
- [ ] The unit test `test_inline_umi_rejected` is replaced with a test that
      inline mode is accepted and produces reads with UMI-length prefix.
- [ ] When `umi.inline: true`, `ReadPair` carries `inline_prefix_r1` and
      `inline_prefix_r2` (Vec<u8>) containing the [UMI][spacer] bytes.
- [ ] FASTQ writer prepends `inline_prefix_r1` (with phred 37 quality bytes)
      to R1 and `inline_prefix_r2` to R2.
- [ ] BAM writer soft-clips the inline prefix region in CIGAR and emits the
      full sequence including the prefix.
- [ ] For duplex AB strand: r1 prefix = UMI_A + spacer, r2 prefix = UMI_B + spacer.
- [ ] For duplex BA strand: r1 prefix = UMI_B + spacer, r2 prefix = UMI_A + spacer.
- [ ] For non-duplex: both r1 and r2 prefixes = UMI + spacer.
- [ ] `fill_umi` and presets compile cleanly with the new field.

## Notes

- UMI is currently embedded in the read name (`read:UMI:AAAA-BBBB`). Keep that
  behaviour. The inline prefix is additional.
- Quality for the UMI prefix: phred 37 (ASCII `F`). Spacer: same.
- `inject_umi_errors` is wired in T148; T146 does not need to call it.
