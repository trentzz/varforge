\section{Results}
\label{sec:results}

\subsection{Setup}

All benchmarks ran on an AMD Ryzen 5 7640HS (6 cores, 12 threads), 12\,GiB DDR5, NVMe SSD, Debian 13, \texttt{rustc} 1.94.0 release mode.
Synthetic references of 1\,MB and 10\,MB were generated with ${\sim}40$\% GC content.
Each configuration ran three times with different seeds.
Mean values are reported.

\subsection{Coverage scaling}

Figure~\ref{fig:cov-scaling} shows wall time and peak memory as a function of coverage depth on the 1\,MB reference with 100 mutations.
Both scale linearly ($R^2 > 0.999$).
Throughput stabilises at ${\sim}6{,}700$ read pairs/s above 10x on the 1\,MB reference.
The lower rate at 1x reflects startup overhead.
On the 10\,MB reference (Table~\ref{tab:scenarios}), throughput rises to ${\sim}14{,}300$ pairs/s because more parallel chunks are available to fill the Rayon thread pool.

\subsection{Feature overhead}

Figure~\ref{fig:features} shows the incremental cost of each feature at 30x on the 1\,MB reference (${\sim}200$K read pairs).
Variant injection and GC bias are within measurement noise ($<1$\%).
FFPE/oxoG artefacts add 15\% via a per-base damage pass.
UMI simulation dominates at +143\% due to PCR family tracking across cycles.
The combined overhead of +198\% is super-additive: UMI families must carry variant and artefact state through each PCR cycle, compounding the individual costs.

\end{figure}

\subsection{hg38 (chr22) benchmarks}

Figure~\ref{fig:hg38} and Table~\ref{tab:hg38} report performance on chromosome~22 from hg38 (${\sim}51$\,Mbp).
Configs~1--6 run at 5x to confirm correct feature operation on real sequence.
Config~7 runs at 30x to validate throughput and memory at realistic WGS depth.
Config~8 demonstrates BED region filtering: restricting simulation to a 1\,Mbp target enables 200x panel coverage in 85\,s instead of the ${\sim}10$\,min a full-chromosome run would require.
UMI configs (3--4) produce more read pairs than the baseline because PCR family expansion is included in the output.

\textbf{Memory usage.}
Peak RAM ranges from 0.56\,GB (BED panel) to 4.21\,GB (30x WGS), well within a standard workstation's capacity.
The 30x WGS run is the most demanding configuration tested; a full human genome at 30x would require roughly $4\times$ chr22's memory (${\sim}17$\,GB on this machine).
Reducing \texttt{performance.output\_buffer\_regions} from the default of 64 lowers peak RAM proportionally, at the cost of reduced parallelism.

\subsection{Twist duplex sequencing for liquid biopsy}
\label{sec:twist}

Twist Biosciences duplex sequencing attaches unique molecular identifiers to both ends of each double-stranded DNA molecule.
The two complementary strand families share a UMI pair $(\alpha,\beta)$ and $(\beta,\alpha)$.
True somatic variants appear on both strands; strand-specific artefacts (FFPE deamination, oxidative damage) appear on only one.
This error-suppression property makes duplex sequencing the most sensitive available approach for low-VAF variant detection in liquid biopsy.

\varforge{} simulates this protocol via the Twist-style duplex configuration (Table~\ref{tab:hg38}, config~4).
At 5x coverage on chr22, \varforge{} produces 3.39\,M read pairs in 299\,s using 3.0\,GB RAM.
The output includes paired FASTQ files with correctly formatted UMI tags (compatible with fgbio and HUMID) and a truth VCF listing injected variants.
This dataset can be used to benchmark duplex-aware callers (e.g.\ Mutect2 with duplex mode, VarDict) under controlled VAF and tumour purity conditions without requiring patient samples.

\subsection{Memory efficiency}

Figure~\ref{fig:mem} shows per-pair memory at ${\sim}0.45$\,KB across coverage depths on the 1\,MB reference, confirming that coverage depth alone does not drive memory growth.
On chr22 at 30x, the per-pair figure rises to ${\sim}0.85$\,KB because the larger chromosome produces more pairs per region chunk, increasing the in-flight buffer state.
The bound is on concurrent batches rather than total pairs: a full human genome at 30x would require roughly $4\times$ chr22's memory, placing it at ${\sim}17$\,GB on this machine.
Users on memory-constrained systems should reduce \texttt{performance.output\_buffer\_regions} (default 64) to lower peak RAM proportionally.

\subsection{Downstream variant calling}
\label{sec:vc}

To confirm that VarForge-generated data exposes real caller behaviour, we ran a four-scenario benchmark using bwa-mem2 \citep{bwamem2} for alignment and Mutect2 \citep{mutect2} for somatic variant calling, evaluated against the VarForge truth VCF using rtg-tools vcfeval.
Stock Mutect2 settings were used throughout; no panel of normals, no ctDNA mode, no artefact filtering.

Figure~\ref{fig:pr-vc} shows where each scenario falls in precision-recall space.
Table~\ref{tab:vc} summarises sensitivity, precision, and F1.
The SNV baseline (30x, 80\% purity) achieves F1\,=\,0.857, consistent with published Mutect2 performance on high-purity tumours.
The panel scenario (200x, 10\% purity) achieves perfect sensitivity (1.000) and precision 0.877 despite no UMI deduplication, illustrating that depth compensates for missing UMI-aware filtering.

The low-VAF ctDNA scenario (30x, 5\% purity) yields precision 0.045 and 1,377 false positives: without ctDNA-specific settings the caller cannot distinguish true low-VAF signal from noise at this tumour fraction.
The FFPE scenario produces 113,764 false positives (Figure~\ref{fig:fp-vc}): FFPE deamination artefacts at a 2\% damage rate overwhelm stock Mutect2 without a panel of normals or OxoG filtering.
These failures are expected and match the published literature \citep{mutect2}.
They demonstrate that VarForge generates data that is realistic enough to expose the sensitivity and specificity limits of a production-grade caller under controlled conditions.
