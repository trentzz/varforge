# Benchmark Methodology

## Overview

This document describes the experimental design used to characterise VarForge performance
across a range of simulation scenarios. Four complementary experiments are reported:

1. **Main scenario benchmarks** — twelve named configurations covering the breadth of
   VarForge's feature set.
2. **Thread scaling** — wall-clock time as a function of thread count on a single config.
3. **Coverage scaling** — wall-clock time and memory as a function of sequencing depth.
4. **Feature overhead** — incremental cost of each major feature.

## Reference Genomes

Synthetic reference FASTA files were generated by a Python script
(`benchmark_output/gen_refs.py`) rather than using a real genome. This ensures
reproducibility (no licence/download issues) and controls for reference GC content.

| File               | Size   | Chromosomes | GC Content |
|--------------------|--------|-------------|------------|
| `ref_1mb.fa`       | ~1 MB  | 6           | ~40%       |
| `ref_10mb.fa`      | ~10 MB | 8           | ~40%       |
| `ref_50mb.fa`      | ~50 MB | 12          | ~40%       |

Each FASTA is accompanied by a matching `.fai` index.

### Iteration Count

Configurations generating fewer than ~2M read pairs (wall time < 2 minutes) received
three iterations. Larger configs received one iteration due to time and memory constraints
on the 12 GiB test system.

| Config | Iterations | Reason for reduction |
|--------|------------|----------------------|
| 01_baseline | 3 | — |
| 02_with_variants | 3 | — |
| 03_high_coverage | 1 | 10MB × 100x → 3.4M reads, 11.6 GB RAM |
| 04_very_high_coverage | 3 | — |
| 05_umi_simplex | 1 | 10MB × 100x + UMI → 10.1M reads, 35 GB swap |
| 06_umi_duplex | 1 | as above |
| 07_cfdna | 3 | — |
| 08_ffpe_artifacts | 3 | — |
| 09_all_features | 1 | 10MB × 100x + all features → 11.1M reads, 40 GB swap |
| 10_large_genome | 1 | 50MB × 30x → 5M reads, 17 GB RAM |
| 11_panel_simulation | 3 | — |
| 12_ultra_deep | 1 | 1MB × 500x → 5M reads, 18 GB RAM |

## Timing

Wall-clock time and peak RSS (resident set size) were captured using
`busybox time -v -o <file>` (BusyBox v1.32.1).  GNU `/usr/bin/time` was not
available on the test system; the BusyBox implementation produces identical output
format and uses the same `wait4`/`getrusage` syscalls.

A dedicated timing file is written for every run:

```
benchmark_output/timing/<config>_iter<N>.txt
```

## Reproducibility

- All runs pass `--seed N` where N equals the iteration number (1, 2, 3).
- The rayon thread pool is set to all available cores (12) for main benchmarks.
- Thread-scaling runs pass `-t <N>` to override the pool size.
- Coverage-scaling runs pass `--coverage <N>` to override the YAML value.

## Coverage Scaling Experiment

Uses a dedicated 1 MB reference at coverages 1x, 5x, 10x, 30x, 50x, 100x, 200x.
A 1 MB reference is used (rather than 10 MB) to keep individual run times short
(<60 s) and peak memory below 3 GB at all tested depths.

## Feature Overhead Experiment

Uses a 1 MB reference at 30x.  Each sub-experiment adds exactly one feature class
above a common base config:

| Label | Features enabled |
|-------|-----------------|
| `feat_base` | none (baseline) |
| `feat_variants` | +random mutations (500 SNV/indel/MNV) |
| `feat_umi` | +UMI simplex inline |
| `feat_artifacts` | +FFPE + oxoG + duplicate inflation |
| `feat_gc_bias` | +GC-content coverage bias |
| `feat_all` | all of the above combined |

## Data Collection Script

The full benchmark suite is implemented in `benchmark_output/run_benchmarks.sh`.
Results are compiled into `benchmark_output/results.json` by
`benchmark_output/compile_results.py`.

Output directories for individual runs are deleted after timing data is extracted
to preserve disk space.
